## Overview

This document describes the 4 transformation steps applied to
retail POS data as it moves from the staging layer to the curated
layer in the `retail_daily_ingestion` pipeline.

Challenge File, Part 3: "Create Python/pandas Transformation Scripts"

**Transformation flow:**
```
staging.sales_transactions (10,050 rows — includes bad rows)
        |
        v
Transform 1: Filter         → removes 50 invalid rows
        |                      writes bad rows to staging.error_log
        v

        |                       total_with_tax, load_timestamp
```

## Transform 1 — Filter Invalid Records

**Source:** Challenge File, Part 3: "Filter Transform"  
**Implemented in:** `transforms.py → filter_invalid_rows()`  
**Called from:** `transform_sales` PythonOperator in DAG

### Rules Applied:
| Field | Rule | Invalid Condition |
|---|---|---|
| transaction_id | Must not be null | `transaction_id IS NULL` |
| quantity | Must be > 0 | `quantity <= 0` |
| unit_price | Must be > 0 | `unit_price <= 0` |

### Code:
```python
bad_mask = ~(
    df["transaction_id"].notna() &
    (df["quantity"].astype(float) > 0) &
    (df["unit_price"].astype(float) > 0)
)
bad_rows = df[bad_mask].copy()
df_clean = df[~bad_mask].copy()
```

### Error Log Handling:
Bad rows are written to `staging.error_log` BEFORE filtering.
Each bad row is captured with:
- `source_table`: `staging.sales_transactions`
- `raw_data`: full row as JSON string
- `error_message`: description of which rule failed
- `logged_at`: timestamp (auto-generated by PostgreSQL DEFAULT)

### Verified Results:
- Input rows: 10,050 (including 50 intentional bad rows)
- Bad rows captured in `staging.error_log`: 50 per run
- Valid rows passed to Transform 2: 10,000

### Sample error_log output:

## Transform 2 — Derived Columns

**Source:** Challenge File, Part 3: "Derived Column Transform"  
**Implemented in:** `transforms.py → apply_derived_columns()`  
**Called from:** `transform_sales` PythonOperator in DAG

### Columns Added:

| Column | Formula | Example |
|---|---|---|
| line_total | `quantity × unit_price × (1 - discount_pct / 100)` | 5 × 200.00 × (1 - 10/100) = 900.00 |
| tax_amount | `line_total × 0.08` | 900.00 × 0.08 = 72.00 |
| total_with_tax | `line_total + tax_amount` | 900.00 + 72.00 = 972.00 |
| load_timestamp | `pd.Timestamp.now()` | 2026-04-08 06:47:35 |
| source_file | `sales_transactions_{run_date}.csv` | sales_transactions_2026-04-07.csv |

### Code:
```python
# Challenge File, Part 3 — exact formulas from challenge
df["line_total"] = (
    df["quantity"] * df["unit_price"] * (1 - df["discount_pct"] / 100)
).round(2)
df["tax_amount"]     = (df["line_total"] * 0.08).round(2)
df["total_with_tax"] = (df["line_total"] + df["tax_amount"]).round(2)
df["load_timestamp"] = pd.Timestamp.now()
df["source_file"]    = f"sales_transactions_{run_date}.csv"
```

### Design Notes:
- Numeric columns explicitly cast to `int`/`float` before calculation
  — CSV ingestion may produce string types
- All financial values rounded to 2 decimal places
- `source_file` provides data lineage — tracks which file produced each row
- `load_timestamp` enables incremental load auditing

## Transform 3 — Daily Aggregation

**Source:** Challenge File, Part 3: "Aggregate Transform"  
**Implemented in:** `transforms.py → aggregate_daily_summary()`  
**Called from:** `transform_sales` PythonOperator in DAG

### Grouping Keys:
- `store_id` — which store
- `product_id` — which product
- `transaction_date` — which day

### Aggregations:

| Output Column | Source Column | Function | Purpose |
|---|---|---|---|
| total_quantity | quantity | SUM | Total units sold |
| total_revenue | line_total | SUM | Total net revenue after discount |
| total_tax | tax_amount | SUM | Total tax collected |
| transaction_count | transaction_id | COUNT | Number of transactions |
| avg_discount | discount_pct | MEAN | Average discount applied |

### Code:
```python
# Challenge File, Part 3 — exact groupby from challenge
daily_summary = df.groupby(
    ["store_id", "product_id", "transaction_date"]
).agg(
    total_quantity    =("quantity",       "sum"),
    total_revenue     =("line_total",     "sum"),
    total_tax         =("tax_amount",     "sum"),
    transaction_count =("transaction_id", "count"),
    avg_discount      =("discount_pct",   "mean"),
).reset_index()
```

### Verified Results:
- Input: 10,000 valid transaction rows
- Output: 6,361 store-product-date aggregated rows
- Compression ratio: 10,000 → 6,361 (36.4% reduction)
- All 50 stores represented
- All 5 product categories represented

## Transform 4 — Product Catalog Enrichment (Lookup)

**Source:** Challenge File, Part 3: "Lookup Transform"  
**Implemented in:** `transforms.py → enrich_with_product_catalog()`  
**Called from:** `transform_sales` PythonOperator in DAG

### Purpose:
Joins the aggregated daily summary with `staging.product_catalog`
to add category, subcategory, and brand attributes to each row.
This enables category-level reporting in the curated layer.

### Join Type: LEFT JOIN
A left join is used deliberately — if a transaction references a
product_id not in the catalog, the transaction row is retained
with NULL category/subcategory/brand rather than being silently
dropped. This prevents data loss and triggers a warning log.

### Code:
```python
# Challenge File, Part 3 — exact query from challenge
product_df = pg_hook.get_pandas_df(
    "SELECT product_id, category, subcategory, brand "
    "FROM staging.product_catalog;"
)
enriched = daily_summary.merge(product_df, on="product_id", how="left")
```

### Columns Added:

| Column | Source | Example Value |
|---|---|---|
| category | staging.product_catalog | Electronics |
| subcategory | staging.product_catalog | Phones |
| brand | staging.product_catalog | Samsung |

## Staging → Curated Load Pattern

**Source:** Challenge File, Part 3:
"Load (upsert into curated.fact_daily_sales)"

After all 4 transforms complete, the enriched DataFrame is:
1. Written to `staging.fact_daily_sales_stage` via
   `pg_hook.insert_rows()` in `transform_sales` PythonOperator
2. UPSERTed into `curated.fact_daily_sales` via
   `load_curated` PostgresOperator

### UPSERT SQL (executed by PostgresOperator):
```sql
INSERT INTO curated.fact_daily_sales (
    store_id, product_id, transaction_date,
    category, subcategory, brand,
    total_quantity, total_revenue, total_tax,
    transaction_count, avg_discount, load_timestamp
)
SELECT
    store_id, product_id, transaction_date::DATE,
    category, subcategory, brand,
    total_quantity, total_revenue, total_tax,
    transaction_count, avg_discount, load_timestamp
FROM staging.fact_daily_sales_stage
ON CONFLICT (store_id, product_id, transaction_date)
DO UPDATE SET
    total_quantity    = EXCLUDED.total_quantity,
    total_revenue     = EXCLUDED.total_revenue,
    total_tax         = EXCLUDED.total_tax,
    transaction_count = EXCLUDED.transaction_count,
    avg_discount      = EXCLUDED.avg_discount,
    load_timestamp    = EXCLUDED.load_timestamp;
```

### Idempotency:
- PRIMARY KEY `(store_id, product_id, transaction_date)` prevents
  duplicates
- ON CONFLICT DO UPDATE ensures reruns update existing rows
- Previous days' data is never affected
